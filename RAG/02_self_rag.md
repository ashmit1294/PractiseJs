|---|---|

## ELI5

Traditional RAG always opens the textbook before answering, even for questions you already know. **Self-RAG** trains the model to continuously ask itself: *"Do I actually need to look this up?"* — and if yes, *"Is what I found actually helpful?"* — and if yes, *"Does my answer match what I found?"*. It's like a student who knows when to use their notes and when to just answer from memory — and also checks their own work as they write.

---

## The 3 Problems Self-RAG Was Built to Fix

### Problem 1 — Indiscriminate Retrieval
Traditional RAG retrieves documents **even when the LLM's own parametric knowledge could answer directly**. Asking *"How many seconds are in a minute?"* doesn't need retrieval — but a RAG chatbot retrieves anyway, injecting noisy, irrelevant context that actually makes the LLM **less confident and less accurate**.

### Problem 2 — Blind Trust in Retrieved Documents
Traditional RAG passes all retrieved documents to the LLM **without checking if they are actually relevant**. A query like *"What causes diabetes?"* might retrieve a document about *effects of diabetes* (high embedding similarity, wrong semantic match). The LLM blindly uses that irrelevant context and generates a wrong answer.

### Problem 3 — No Self-Verification
After Traditional RAG generates an answer, **there is zero post-generation check**. The answer goes straight to the user — whether hallucinated, partially fabricated, or factually correct — with no reflection on its own quality.

**Self-RAG fixes all three** by making the LLM actively judge its own retrieval decisions and generated answers at every step via four self-reflection checkpoints.

---

## Core Concept — Reflection Tokens

Self-RAG fine-tunes a single LLM to emit four special control tokens **interleaved with normal generation**:

| Token | Type | Values | Purpose |
|---|---|---|---|
| `[Retrieve]` | Retrieval decision | `Yes` / `No` / `Continue` | Should I fetch evidence right now? |
| `[IsRel]` | Relevance assessment | `Relevant` / `Irrelevant` | Is this retrieved passage actually relevant to my query? |
| `[IsSup]` | Support assessment | `Fully supported` / `Partially` / `No support` | Does my generated output match the retrieved passage? |
| `[IsUse]` | Utility assessment | `5` / `4` / `3` / `2` / `1` | How useful is my overall response to the user? |

These tokens are **generated by the same model** — not a separate classifier. The model learns when to emit them during fine-tuning.

> **Practical vs Paper approach:** The original Self-RAG paper requires a fine-tuned model that natively emits these tokens. In practice (e.g., with GPT-4o or Claude), the same self-reflection logic is implemented via LangGraph — each reflection check becomes a separate LLM node call using structured output (Pydantic schema). Both approaches are documented below.

## Architecture: Fine-Tuned Model Approach (Paper)

```
User Query
    ↓
LLM generates text + emits [Retrieve=Yes/No]
    │
    ├── [Retrieve=No]
    │       ↓
    │   Continue generating → emit [IsUse] score
    │       ↓
    │   Final answer (no retrieval cost)
    │
    └── [Retrieve=Yes]
            ↓
        Retrieve k passages from Vector DB
            ↓
        For each passage p_i:
            LLM generates continuation → emits [IsRel] + [IsSup]
            Segment-level score = [IsRel] × [IsSup] × [IsUse]
            ↓
        Beam search: select best passage by segment score
            ↓
        Continue generation with best passage as context
            ↓
        Emit [IsUse] for final answer
            ↓
        If [IsUse] < threshold → re-retrieve with refined query
            ↓
        Final answer
```

---

## Segment-Level Beam Search (Key Mechanism)

Unlike Traditional RAG which retrieves once, Self-RAG scores **multiple candidate continuations** in parallel — one per retrieved passage — and selects the best at each segment boundary.

```
Query: "What causes Aurora Borealis?"

Retrieved passages: [p1, p2, p3]

For p1: LLM generates segment s1 → [IsRel=Relevant] [IsSup=Fully supported] → score = 1.0
For p2: LLM generates segment s2 → [IsRel=Relevant] [IsSup=Partially]      → score = 0.5
For p3: LLM generates segment s3 → [IsRel=Irrelevant]                       → score = 0.0

Select: p1's continuation (score = 1.0) → proceed with that generation path
```

This is **beam search at the segment level** — not over tokens, but over full sentence/paragraph segments, each grounded in a different retrieved passage.

## Architecture: LangGraph Practical Approach (7-Node Graph)

When using any standard LLM (GPT-4o, Claude, Gemini) without fine-tuning, Self-RAG is implemented as a **LangGraph state machine** where each Self-RAG reflection token maps to a dedicated node:

```
User Query
    ↓
[Node 1: Decide Retrieval]   ← Is retrieval even needed?
    │
    ├── NO  ──→  [Direct LLM Generate]  ──→  END
    │
    └── YES
          ↓
    [Node 2: Retrieve]         ← Fetch top-k from vector store
          ↓
    [Node 3: Is Relevant?]     ← Filter each doc (True/False per doc)
          │
          ├── No relevant docs found
          │         ↓
          │   [No Relevant Docs Node]  ──→  (Web Search / END)
          │
          └── Relevant docs found
                    ↓
          [Node 4: Generate from Context]   ← LLM answers from filtered docs only
                    ↓
          [Node 5: Is Supported?]    ← Hallucination check
          (fully / partially / not_supported)
                    │
                    ├── not fully supported
                    │         ↓
                    │   [Node 6: Revise Answer]  ←──────┐
                    │         ↓                          │
                    │   back to Is Supported  ───────────┘  (max_retries loop)
                    │
                    └── fully_supported
                              ↓
                    [Node 7: Is Useful?]   ← Does it answer the question?
                              │
                              ├── YES  ──→  [Finalize]  ──→  END
                              └── NO   ──→  [No Answer Found]  ──→  END
                                           (full Self-RAG: rewrite query + re-retrieve)
```

### LangGraph State Schema

The shared state object passed through all nodes:

| Field | Type | Purpose |
|---|---|---|
| `question` | `str` | Original user query |
| `need_retrieval` | `bool` | Output of Node 1 |
| `docs` | `list[Document]` | All retrieved documents from vector store |
| `relevant_docs` | `list[Document]` | Filtered relevant docs (output of Node 3) |
| `context` | `str` | Merged text from `relevant_docs` (input to Node 4) |
| `answer` | `str` | Current generated answer (may be revised) |
| `is_support` | `str` | `fully_supported` / `partially_supported` / `not_supported` |
| `evidence` | `list[str]` | Facts extracted from context (for debugging) |
| `is_use` | `str` | `useful` / `not_useful` |
| `reason` | `str` | Why useful or not |
| `retries` | `int` | Retry counter — prevents infinite hallucination loops |

### Node-by-Node Breakdown

**Node 1 — Decide Retrieval**  
Uses a `ShouldRetrieve` Pydantic schema with a boolean `retrieve` field. System prompt instructs: return `True` if specific facts or citations are needed, `False` for general explanations the LLM already knows. Uses `with_structured_output()` for deterministic JSON.

**Node 2 — Retrieve**  
Simple `retriever.invoke(question)` — fetches top-k chunks from vector store. Built with `RecursiveCharacterTextSplitter` (chunk_size=600, overlap=150, OpenAI embeddings).

**Node 3 — Is Relevant?**  
Loops over **every retrieved doc individually** and asks an LLM judge (`IsRelevant` Pydantic schema) whether that specific document helps answer the question. Only `True` docs flow to `relevant_docs`. Irrelevant docs are silently dropped at this stage.

**Node 4 — Generate from Context**  
Merges all `relevant_docs` into a single `context` string. Strict system prompt: *"Answer using ONLY the provided context."* If zero relevant docs → routes to No Relevant Docs node (replaceable with a web-search node for the Incorrect path).

**Node 5 — Is Supported? (Hallucination Check)**  
The most critical node. A grader LLM receives `question + generated_answer + context` and classifies:

| Label | Meaning |
|---|---|
| `fully_supported` | Every fact in the answer comes from the retrieved docs |
| `partially_supported` | Some facts from docs, some LLM-fabricated |
| `not_supported` | All facts hallucinated — no grounding at all |

**Node 6 — Revise Answer (Hallucination Loop)**  
If `partially_supported` or `not_supported`: strict system prompt *"Remove all facts not present in the context."* Revised answer loops back to Node 5. `retries` counter enforces a max (e.g., 5) to prevent infinite loops.

**Node 7 — Is Useful?**  
Even a fully-supported, non-hallucinated answer can be useless — factually correct but doesn't address the user's actual question. LLM asked: *"Does this answer actually address the question?"* If `not_useful` → currently routes to "No Answer Found". In full Self-RAG, this triggers query rewrite + re-retrieval loop.

### Self-Reflection Loops (with Max-Retry Guards)

```
Loop 1 — Hallucination Loop:
  Node 5 (Is Supported) → Node 6 (Revise) → Node 5
  Break condition: is_support == 'fully_supported' OR retries >= MAX_RETRIES

Loop 2 — Usefulness Loop (full Self-RAG):
  Node 7 (Is Useful) → Rewrite Query → Node 2 (Retrieve) → ... → Node 7
  Break condition: is_use == 'useful' OR retries >= MAX_RETRIES
```

Both loops **must be guarded by a retry counter** — without it, a pathological query can infinite-loop between retrieval and revision.

### Python Implementation with LangGraph + Pydantic

```python
from typing import TypedDict, List
from langchain_core.documents import Document
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_community.vectorstores import FAISS
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langgraph.graph import StateGraph, END
from pydantic import BaseModel, Field

# ── State ──────────────────────────────────────────────────────────
class SelfRAGState(TypedDict):
    question: str
    need_retrieval: bool
    docs: List[Document]
    relevant_docs: List[Document]
    context: str
    answer: str
    is_support: str    # fully_supported | partially_supported | not_supported
    is_use: str        # useful | not_useful
    retries: int

# ── Pydantic Schemas (structured output) ──────────────────────────
class ShouldRetrieve(BaseModel):
    retrieve: bool = Field(description="True if specific facts/citations needed, False for general knowledge")

class IsRelevant(BaseModel):
    relevant: bool = Field(description="True if document helps answer the question")

class SupportScore(BaseModel):
    support: str = Field(description="fully_supported | partially_supported | not_supported")

class UseScore(BaseModel):
    useful: bool = Field(description="True if answer addresses the user question")

llm = ChatOpenAI(model="gpt-4o", temperature=0)

# ── Node 1: Decide Retrieval ───────────────────────────────────────
def decide_retrieval(state: SelfRAGState):
    grader = llm.with_structured_output(ShouldRetrieve)
    result = grader.invoke([
        {"role": "system", "content": "Decide if retrieval is needed. Return True for specific facts/citations."},
        {"role": "user", "content": state["question"]},
    ])
    return {"need_retrieval": result.retrieve}

# ── Node 2: Retrieve ───────────────────────────────────────────────
def retrieve(state: SelfRAGState):
    docs = retriever.invoke(state["question"])
    return {"docs": docs}

# ── Node 3: Is Relevant? ───────────────────────────────────────────
def grade_relevance(state: SelfRAGState):
    grader = llm.with_structured_output(IsRelevant)
    relevant_docs = []
    for doc in state["docs"]:
        result = grader.invoke([
            {"role": "system", "content": "Is this document relevant to the question?"},
            {"role": "user", "content": f"Question: {state['question']}\nDocument: {doc.page_content}"},
        ])
        if result.relevant:
            relevant_docs.append(doc)
    return {"relevant_docs": relevant_docs}

# ── Node 4: Generate from Context ─────────────────────────────────
def generate(state: SelfRAGState):
    context = "\n\n---\n\n".join(d.page_content for d in state["relevant_docs"])
    response = llm.invoke([
        {"role": "system", "content": "Answer ONLY using the provided context. Do not use prior knowledge."},
        {"role": "user", "content": f"Context:\n{context}\n\nQuestion: {state['question']}"},
    ])
    return {"answer": response.content, "context": context}

# ── Node 5: Is Supported? ──────────────────────────────────────────
def check_support(state: SelfRAGState):
    grader = llm.with_structured_output(SupportScore)
    result = grader.invoke([
        {"role": "system", "content": "Classify: is the answer fully, partially, or not supported by the context?"},
        {"role": "user", "content": f"Context:\n{state['context']}\nAnswer:\n{state['answer']}"},
    ])
    return {"is_support": result.support, "retries": state.get("retries", 0) + 1}

# ── Node 6: Revise Answer ──────────────────────────────────────────
def revise_answer(state: SelfRAGState):
    response = llm.invoke([
        {"role": "system", "content": "Revise the answer. Remove ALL facts not present in the context."},
        {"role": "user", "content": f"Context:\n{state['context']}\nAnswer to revise:\n{state['answer']}"},
    ])
    return {"answer": response.content}

# ── Node 7: Is Useful? ─────────────────────────────────────────────
def check_usefulness(state: SelfRAGState):
    grader = llm.with_structured_output(UseScore)
    result = grader.invoke([
        {"role": "system", "content": "Does this answer actually address the user's question?"},
        {"role": "user", "content": f"Question: {state['question']}\nAnswer: {state['answer']}"},
    ])
    return {"is_use": "useful" if result.useful else "not_useful"}

# ── Graph Assembly ─────────────────────────────────────────────────
MAX_RETRIES = 5

graph = StateGraph(SelfRAGState)
graph.add_node("decide_retrieval",  decide_retrieval)
graph.add_node("retrieve",          retrieve)
graph.add_node("grade_relevance",   grade_relevance)
graph.add_node("generate",          generate)
graph.add_node("check_support",     check_support)
graph.add_node("revise_answer",     revise_answer)
graph.add_node("check_usefulness",  check_usefulness)

graph.set_entry_point("decide_retrieval")

# Edge: retrieve or go direct
graph.add_conditional_edges("decide_retrieval", lambda s: "retrieve" if s["need_retrieval"] else END)

graph.add_edge("retrieve", "grade_relevance")

# Edge: any relevant docs?
graph.add_conditional_edges("grade_relevance", lambda s: "generate" if s["relevant_docs"] else END)

graph.add_edge("generate", "check_support")

# Hallucination loop with retry guard
graph.add_conditional_edges(
    "check_support",
    lambda s: "check_usefulness" if s["is_support"] == "fully_supported" or s["retries"] >= MAX_RETRIES
              else "revise_answer"
)
graph.add_edge("revise_answer", "check_support")

# Usefulness gate
graph.add_conditional_edges("check_usefulness", lambda s: END)

app = graph.compile()
```

## Implementation with LangGraph

LangGraph models the Self-RAG flow as a state machine — each node is a step, edges are reflection token routes:

```javascript
// Self-RAG LangGraph flow (Node.js / TypeScript)
import { StateGraph, END } from '@langchain/langgraph';

const selfRagGraph = new StateGraph({
  channels: {
    query:          { value: null },
    retrievedDocs:  { value: [] },
    generation:     { value: null },
    reflectTokens:  { value: {} },
    finalAnswer:    { value: null },
  },
});

// Node 1: Decide whether to retrieve
selfRagGraph.addNode('decideRetrieve', async (state) => {
  const output = await selfRagModel.generate(state.query);
  const shouldRetrieve = output.includes('[Retrieve=Yes]');
  return { reflectTokens: { retrieve: shouldRetrieve }, generation: output };
});

// Node 2: Retrieve k passages
selfRagGraph.addNode('retrieve', async (state) => {
  const docs = await vectorStore.similaritySearch(state.query, 5);
  return { retrievedDocs: docs };
});

// Node 3: Score passages + generate with best
selfRagGraph.addNode('generateWithPassage', async (state) => {
  const scored = await Promise.all(
    state.retrievedDocs.map(async (doc) => {
      const output = await selfRagModel.generate(
        `Context: ${doc.pageContent}\n\nQuery: ${state.query}`
      );
      const isRel  = output.includes('[IsRel=Relevant]')      ? 1.0 : 0.0;
      const isSup  = output.includes('[IsSup=Fully supported]') ? 1.0
                   : output.includes('[IsSup=Partially]')       ? 0.5 : 0.0;
      return { doc, output, score: isRel * isSup };
    })
  );

  const best = scored.sort((a, b) => b.score - a.score)[0];
  return { finalAnswer: best.output, reflectTokens: { isUse: best.score } };
});

// Node 4: Evaluate utility — re-retrieve if low
selfRagGraph.addNode('evaluateUtility', async (state) => {
  if (state.reflectTokens.isUse < 0.5) {
    // Refine query and go back to retrieve
    return { query: `More specific: ${state.query}` };
  }
  return { finalAnswer: state.finalAnswer };
});

// Conditional edges based on reflection tokens
selfRagGraph.addConditionalEdges('decideRetrieve', (state) =>
  state.reflectTokens.retrieve ? 'retrieve' : END
);
selfRagGraph.addEdge('retrieve', 'generateWithPassage');
selfRagGraph.addEdge('generateWithPassage', 'evaluateUtility');
selfRagGraph.addConditionalEdges('evaluateUtility', (state) =>
  state.reflectTokens.isUse < 0.5 ? 'retrieve' : END
);

selfRagGraph.setEntryPoint('decideRetrieve');
const app = selfRagGraph.compile();
```

## Fine-Tuning for Self-RAG

Self-RAG requires a **specially trained model** — you cannot use a vanilla GPT-4o or Claude. The model must be fine-tuned to:
1. Generate reflection tokens at the right positions
2. Understand what `[Retrieve]`, `[IsRel]`, `[IsSup]`, `[IsUse]` mean
3. Use beam search at segment level

**Training data format:**

```
Input:  "What is photosynthesis?"
Output: "[Retrieve=Yes] Photosynthesis is [IsRel=Relevant] [IsSup=Fully supported] the process 
         by which plants convert sunlight into energy. [IsUse=5]"

Input:  "What is 2+2?"
Output: "[Retrieve=No] The answer is 4. [IsUse=5]"
```

Base models commonly fine-tuned: **Llama-3-8B**, **Llama-3-70B**, **Mistral-7B**.  
Pre-trained Self-RAG models are available on HuggingFace: `selfrag/selfrag_llama2_7b`.

## Self-RAG vs Traditional RAG

| Dimension | Traditional RAG | Self-RAG |
|---|---|---|
| **Retrieval trigger** | Always (every query) | Conditional on `[Retrieve]` token |
| **Retrieval cost** | 100% of queries | ~30–60% of queries (lower cost) |
| **Passage scoring** | None (all injected) | Per-passage `[IsRel]` + `[IsSup]` scoring |
| **Beam search** | No | Yes — segment-level candidate selection |
| **Fine-tuning required** | No | Yes — model must learn reflection tokens |
| **Hallucination control** | Via grounding instruction only | Via `[IsSup]` self-verification per passage |
| **Latency** | Fixed (always retrieves) | Variable (lower on average) |
| **Best for** | Plug-and-play with any LLM | High-volume, latency-sensitive, hallucination-critical |

## Production Scaling Challenges

| Challenge | Solution |
|---|---|
| **Fine-tuned model is large** | Use 8B param model with 4-bit quantisation (bitsandbytes / GPTQ) on GPU |
| **Beam search multiplies GPU compute** | Limit beam width to k=3–5; use async parallel scoring |
| **Reflection token generation is sequential** | Use speculative decoding or smaller draft model for token prediction |
| **Re-retrieval loops** | Cap max re-retrieve iterations at 2–3 to prevent infinite loops |
| **Model drift after fine-tuning updates** | Version model artifacts; shadow-deploy new versions |
| **Node 3/5/7 bottleneck at scale** | `IsRelevant`, `IsSupported`, `IsUseful` each make individual LLM calls per doc/answer — at millions of documents these become the compute bottleneck; fix via **async batching** and **LLM call parallelisation** |
| **Reflection graders are expensive per-doc** | Use a smaller grader model (GPT-4o-mini, Llama-3-8B) for Node 3 and Node 7; only use full model for Node 5 (hallucination check) |

## Interview Questions

**Q: What is the fundamental difference between Self-RAG and Traditional RAG?**  
A: Traditional RAG always retrieves, regardless of whether retrieval helps. Self-RAG trains the LLM itself to decide when to retrieve (via `[Retrieve]`), whether what was retrieved is relevant (`[IsRel]`), whether its answer is supported by what it retrieved (`[IsSup]`), and whether its answer is useful (`[IsUse]`). The key shift is from external orchestration logic to internal model decision-making.

**Q: Can you use GPT-4o for Self-RAG?**  
A: Not natively. GPT-4o hasn't been fine-tuned to emit Self-RAG reflection tokens. You can simulate the pattern by using GPT-4o as an evaluator with explicit prompts (asking it "Is the retrieved passage relevant?"), but that is CRAG-style separate evaluation, not true Self-RAG. True Self-RAG requires a fine-tuned model.

**Q: What is segment-level beam search in Self-RAG?**  
A: Instead of scoring the entire answer at once, Self-RAG scores the answer segment by segment — one segment per retrieved passage. The highest-scoring segment (based on `[IsRel]` × `[IsSup]`) determines which passage's generation path to follow. This is beam search at the output segment level, not at the token level.

**Q: How does Self-RAG reduce hallucination?**  
A: The `[IsSup]` token forces the model to verify: "Is my generated claim actually supported by the retrieved passage?" If the score is low (Partially or No support), the generation is discarded or the passage is swapped. Unlike Traditional RAG where the model can ignore retrieved context, Self-RAG self-penalises unsupported claims during generation.
